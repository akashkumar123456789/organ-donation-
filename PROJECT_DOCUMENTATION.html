Abstract

The Organ Donation Management System is a comprehensive web-based application designed to streamline and optimize the process of organ donation and transplantation. This system addresses the critical need for efficient management of organ donors, patients awaiting transplants, hospital facilities, and the matching process between donors and recipients.

The application implements a complete CRUD (Create, Read, Update, Delete) architecture across five main modules: Donor Profiles, Patient Waitlist, Hospital Inventory, Organ Match Tracker, and Medical Records Log. Built using Python 3.8+ for the backend with SQLite3 database, and modern HTML5, CSS3, and JavaScript ES6+ for the frontend, the system provides a robust and user-friendly interface for managing all aspects of organ donation operations.

Key features include real-time donor-patient matching based on blood group and organ type compatibility, urgency-based patient prioritization, hospital capacity management, comprehensive medical record tracking with follow-up notes, and data export capabilities in CSV and PDF formats. The system employs a RESTful API architecture with MVC pattern, ensuring scalability and maintainability.

The application features a modern dark-themed user interface with smooth animations, responsive design for all screen sizes, and comprehensive form validation to ensure data integrity. The system automatically populates sample data on first run, making it immediately usable for demonstration and testing purposes.

This project demonstrates the practical application of full-stack web development technologies in solving real-world healthcare challenges. The system has the potential to significantly improve the efficiency of organ donation operations, reduce waiting times for patients, and ultimately save lives through better coordination and management of the organ donation process.

Keywords: Organ Donation, Healthcare Management, Web Application, CRUD Operations, RESTful API, SQLite Database, Real-time Matching, Medical Records

Chapter 1
Introduction

1.1 Overview

Organ donation and transplantation represent one of the most significant medical advances of the modern era, offering hope and life-saving treatment to thousands of patients suffering from end-stage organ failure. However, the process of managing organ donations, maintaining donor and recipient databases, coordinating between hospitals, and ensuring timely matching of organs remains a complex and challenging task.

The Organ Donation Management System is a comprehensive web-based application designed to address these challenges by providing a centralized platform for managing all aspects of organ donation operations. The system facilitates efficient coordination between donors, patients, hospitals, and medical professionals, ultimately aiming to reduce waiting times and save more lives.

This application implements a modern full-stack architecture with a Python-based backend, SQLite database for data persistence, and a responsive frontend built with HTML5, CSS3, and JavaScript. The system provides complete CRUD (Create, Read, Update, Delete) functionality across five main modules, ensuring comprehensive management of the entire organ donation lifecycle.

1.2 Problem Statement

The current organ donation process faces several critical challenges:

  • Manual Record Management: Many facilities still rely on paper-based or fragmented digital systems, leading to inefficiencies and potential errors in record-keeping.
  • Delayed Matching Process: The time-sensitive nature of organ transplantation requires rapid matching between donors and recipients, which is often hindered by inadequate systems.
  • Lack of Centralized Information: Information about available donors, waiting patients, and hospital capacities is often scattered across multiple systems, making coordination difficult.
  • Urgency Prioritization: Determining which patients should receive priority based on medical urgency requires a systematic approach that many existing systems lack.
  • Follow-up Tracking: Post-transplant follow-up and long-term outcome tracking are essential but often poorly managed.
  • Data Accessibility: Medical professionals need quick access to comprehensive patient and donor information, which is not always readily available.

1.3 Objectives

The primary objectives of the Organ Donation Management System are:

  1. Centralized Database Management: Create a unified database system for storing and managing information about donors, patients, hospitals, matches, and medical records.
  2. Efficient Matching Algorithm: Implement a system for matching donors with compatible recipients based on blood group, organ type, and urgency levels.
  3. Real-time Updates: Provide real-time updates on donor availability, patient status, and hospital capacity to facilitate quick decision-making.
  4. Comprehensive CRUD Operations: Enable complete Create, Read, Update, and Delete operations across all modules to ensure data accuracy and flexibility.
  5. User-Friendly Interface: Design an intuitive and responsive user interface that can be easily used by medical professionals with varying levels of technical expertise.
  6. Data Export and Reporting: Provide functionality to export data in various formats (CSV, PDF) for reporting and analysis purposes.
  7. Form Validation: Implement comprehensive validation to ensure data integrity and prevent errors in critical medical information.
  8. Scalability: Design the system architecture to be scalable and maintainable for future enhancements and increased user load.

1.4 Scope of the Project

The scope of this project encompasses the following key areas:

1.4.1 Functional Scope

  • Donor Management: Registration, profile management, and status tracking of organ donors
  • Patient Management: Waitlist management with urgency-based prioritization
  • Hospital Management: Inventory of medical facilities with capacity and operational status tracking
  • Match Tracking: Real-time tracking of donor-patient matches through various stages (Initiated, Procurement, Surgery, Completed)
  • Medical Records: Comprehensive documentation of surgeries, success rates, and follow-up notes
  • Data Export: CSV and PDF export capabilities for individual records and bulk data

1.4.2 Technical Scope

  • Backend development using Python 3.8+ with HTTP server
  • Database implementation using SQLite3 with relational schema
  • Frontend development using HTML5, CSS3, and JavaScript ES6+
  • RESTful API architecture for client-server communication
  • MVC (Model-View-Controller) design pattern
  • Responsive design for cross-device compatibility

1.4.3 Limitations

  • The system is designed as a single-user application without multi-user authentication
  • Advanced features like email notifications and SMS alerts are not implemented
  • The system does not include integration with external medical databases or government registries
  • Real-time collaboration features for multiple users are not included

Chapter 5
System Design

5.1 System Architecture

The Organ Donation Management System follows a three-tier architecture:

5.1.1 Presentation Layer (Frontend)

The presentation layer handles user interaction and display:

  • HTML Pages: Structure and content (index.html, 404.html)
  • CSS Stylesheets: Visual design and responsive layout (style.css)
  • JavaScript Controllers: Page-specific logic and user interactions
    • dashboardController.js - Overview and statistics
    • donorController.js - Donor management
    • patientController.js - Patient waitlist
    • hospitalController.js - Hospital inventory
    • matchController.js - Match tracking
    • recordController.js - Medical records
  • Utility Scripts:
    • app.js - Main application controller and navigation
    • api.js - API service layer for HTTP requests
    • validation.js - Form validation utilities
    • enhanced-features.js - PDF export and additional features

5.1.2 Application Layer (Backend)

The application layer processes business logic and handles requests:

  • app.py: HTTP server and request handler
  • router.py: API routing and endpoint mapping
  • Controllers: Business logic for each module
    • donor_controller.py - Donor CRUD operations
    • patient_controller.py - Patient CRUD operations
    • hospital_controller.py - Hospital CRUD operations
    • match_controller.py - Match CRUD operations with JOIN queries
    • record_controller.py - Record CRUD operations with complex JOINs
    • export_controller.py - CSV export and dashboard data

5.1.3 Data Layer (Database)

The data layer manages data persistence and retrieval:

  • connection.py: Database initialization and connection management
  • organ_donation.db: SQLite database file with 5 tables

5.1.4 System Flow

Request Flow:

  1. User interacts with frontend (clicks button, submits form)
  2. JavaScript controller captures event and validates input
  3. API service sends HTTP request to backend
  4. HTTP server receives request and passes to router
  5. Router directs request to appropriate controller
  6. Controller executes business logic and database operations
  7. Database returns data to controller
  8. Controller formats response as JSON
  9. HTTP server sends response to frontend
  10. JavaScript updates UI with new data

5.2 Database Design

5.2.1 Database Schema

The system uses five interconnected tables:

Table 1: donors

Column Name Data Type Constraints Description
id INTEGER PRIMARY KEY, AUTOINCREMENT Unique donor identifier
name TEXT NOT NULL Donor full name
blood_group TEXT NOT NULL Blood type (A+, A-, B+, B-, AB+, AB-, O+, O-)
organ_type TEXT NOT NULL Organ available for donation
contact TEXT Contact information

Table 2: patients

Column Name Data Type Constraints Description
id INTEGER PRIMARY KEY, AUTOINCREMENT Unique patient identifier
name TEXT NOT NULL Patient full name
blood_group TEXT NOT NULL Blood type
organ_needed TEXT NOT NULL Organ required for transplant
urgency_level INTEGER DEFAULT 5 Medical urgency (1-10 scale)
contact TEXT Contact information
status TEXT DEFAULT 'Waiting' Patient status (Waiting, Matched, Completed)

Table 3: hospitals

Column Name Data Type Constraints Description
id INTEGER PRIMARY KEY, AUTOINCREMENT Unique hospital identifier
name TEXT NOT NULL Hospital name
location TEXT NOT NULL Hospital location/address
capacity INTEGER DEFAULT 10 Number of available beds/units
operating_status TEXT DEFAULT 'Active' Hospital status (Active, Full, Maintenance)

Table 4: matches

Column Name Data Type Constraints Description
id INTEGER PRIMARY KEY, AUTOINCREMENT Unique match identifier
donor_id INTEGER FOREIGN KEY → donors(id) Reference to donor
patient_id INTEGER FOREIGN KEY → patients(id) Reference to patient
hospital_id INTEGER FOREIGN KEY → hospitals(id) Reference to hospital
stage TEXT DEFAULT 'Initiated' Match stage (Initiated, Procurement, Surgery, Completed)
created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP Match creation timestamp

Table 5: records

Column Name Data Type Constraints Description
id INTEGER PRIMARY KEY, AUTOINCREMENT Unique record identifier
match_id INTEGER FOREIGN KEY → matches(id) Reference to match
surgery_date TEXT Date of surgery
success_status TEXT Surgery outcome (Successful, Pending, etc.)
notes TEXT Surgery notes and observations
follow_up_notes TEXT Post-surgery follow-up information

5.2.2 Entity Relationships

  • donors ↔ matches: One-to-Many (one donor can have multiple matches over time)
  • patients ↔ matches: One-to-Many (one patient can have multiple match attempts)
  • hospitals ↔ matches: One-to-Many (one hospital handles multiple matches)
  • matches ↔ records: One-to-One (each match has one medical record)
